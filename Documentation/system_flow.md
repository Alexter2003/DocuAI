# Complete System Flow

This document details the step-by-step lifecycle of DocuAI from onboarding to documentation generation.

## 1. Onboarding Flow
1. User visits `app.docuai.com`.
2. Clicks "Login with GitHub" (OAuth). 
3. Selects which repositories they want DocuAI to monitor.
4. Next.js calls NestJS backend -> Saves User and Project in Su   pabase.
5. User is instructed to add a `.github/workflows/docuai.yml` file to their repo (or DocuAI opens a PR automatically to add it).

## 2. Execution Flow (The "Action")
1. Developer works on a `feature` branch, makes 20 commits.
2. Developer opens a Pull Request.
3. PR is reviewed and **Merged** into `main`.
4. The `.github/workflows/docuai.yml` workflow triggers inside GitHub servers.
5. The Action:
   - Reads the `.docuai.yml` configuration (e.g., "Ignore /tests, Document in English").
   - Gathers the code diffs for this PR.
   - Pings the AI API (Gemini or BYOK) to generate documentation and Mermaid diagrams.
6. The Action sends a POST request to `api.docuai.com/v1/webhook/doc-generated` with the payload string.
7. The Action posts a summary comment back onto the GitHub PR.

## 3. Storage & Q&A Flow

### 3.1 What gets stored where (and why)

#### Relational DB — PostgreSQL (`documentation_entries` table)
- **What:** The full Markdown document generated by the AI for a given PR.
- **Why:** To display it in the dashboard. When a user opens DocuAI and clicks on a PR, they see this full formatted document.
- **Access pattern:** `SELECT * FROM documentation_entries WHERE repository_id = X ORDER BY generated_at DESC`

#### Vector DB — pgvector (`documentation_chunks` table) — *implemented separately*
- **What:** The sadsme Markdown document, but split into ~500-word chunks, with an embedding (vector of numbers) generated for each chunk.
- **Why:** To power the Q&A chatbot. The AI cannot "read" the full document on every question — it's too slow and expensive. Instead, when a user asks a question, the system finds the 3–5 most *semantically similar* chunks and feeds those to the AI as context.
- **Access pattern:** `SELECT content FROM documentation_chunks ORDER BY embedding <-> $query_embedding LIMIT 5`

> **Key insight:** The relational entry is what the human reads. The vector chunks are what the AI reads.

### 3.2 Step-by-step
1. NestJS receives the webhook payload from GitHub Actions.
2. NestJS saves the full Markdown + metadata (repo, PR number, model used, status) into `documentation_entries` (PostgreSQL).
3. NestJS splits the Markdown into ~500-word chunks.
4. NestJS calls the embedding model (Gemini) to generate a vector for each chunk.
5. NestJS saves each chunk + its vector into `documentation_chunks` (pgvector).
6. Later, a user opens the Next.js Dashboard and uses the Chat Assistant.
7. User asks: "How does the Auth module work?"
8. NestJS converts the question into a vector → runs similarity search against `documentation_chunks` → retrieves the 5 most relevant chunks → sends them as context to Gemini → returns a precise answer.
